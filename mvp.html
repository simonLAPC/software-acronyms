MVP

Model-View-Presenter

An architectural pattern that separates an application's logic into three interconnected components, improving modularity and testability.

  • Separation of Concerns: The View is passive and all logic is in the Presenter.

  • High Testability: The Presenter is decoupled from the UI, making it easy to unit test.

  • Improved Reusability: Swap out the View with minimal changes to the Presenter.

Note: While similar, MVC (Model-View-Controller) has the Controller handle input, but the View can still directly access the Model. In MVVM (Model-View-ViewModel), the ViewModel exposes data streams that the View binds to.

# A simple UI where all logic is mixed in the View.
# (Conceptual example, doesn't require a UI library)
class CounterAppView:
    def __init__(self):
        self.count = 0  # Data logic
        self.display_label = "" # UI state

    def _update_display(self):
        self.display_label = f"Count: {self.count}"
        print(f"UI Updated: {self.display_label}")

    def on_increment_button_click(self):
        # Presentation logic mixed with data logic
        self.count += 1
        self._update_display()

# Hard to test the logic without creating the UI.
app = CounterAppView()
app.on_increment_button_click()

MVP Pattern

Show Code 

class CounterModel:
    def __init__(self): self.count = 0
    def increment(self): self.count += 1

class CounterView: # The passive View
    def update_display(self, text): print(f"UI Updated: {text}")

class CounterPresenter: # The logic hub
    def __init__(self, model, view):
        self.model, self.view = model, view

    def on_increment(self):
        self.model.increment()
        new_text = f"Count: {self.model.count}"
        self.view.update_display(new_text)

# Easily testable and well-structured.
presenter = CounterPresenter(CounterModel(), CounterView())
presenter.on_increment()
