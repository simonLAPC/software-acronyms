SOLID

Dependency Inversion Principle

High-level modules should not depend on low-level modules. Both should depend on abstractions (e.g., interfaces).

  • Loose Coupling: High-level logic isn't tied to specific low-level implementations.

  • Testability: Allows for easy mocking of dependencies during testing.

  • Flexibility: Swap implementations without affecting high-level code.

Violation of DIP

Show Code 

# High-level module (Notifier) depends directly on a low-level module (EmailClient).
class EmailClient:
    def send(self, message):
        print(f"Sending email: {message}")

class Notifier:
    def __init__(self):
        self.client = EmailClient() # Tight coupling!

    def send_notification(self, message):
        self.client.send(message)

# What if we want to send an SMS instead? We have to change Notifier.
notifier = Notifier()
notifier.send_notification("Hello World!")

Adhering to DIP

Show Code 

# Both high- and low-level modules depend on an abstraction.
from abc import ABC, abstractmethod

class NotificationClient(ABC):
    @abstractmethod
    def send(self, message): pass

class EmailClient(NotificationClient):
    def send(self, message): print(f"Email: {message}")

class SmsClient(NotificationClient):
    def send(self, message): print(f"SMS: {message}")

class Notifier:
    def __init__(self, client: NotificationClient):
        self.client = client # Depends on abstraction!

    def send_notification(self, message):
        self.client.send(message)

# We can easily swap implementations.
email_notifier = Notifier(EmailClient())
sms_notifier = Notifier(SmsClient())
email_notifier.send_notification("Hello via Email!")
sms_notifier.send_notification("Hello via SMS!")
