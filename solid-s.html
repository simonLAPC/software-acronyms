SOLID

Single Responsibility Principle

A class or module should have one, and only one, reason to change. This principle encourages you to create small, focused components.

  • Easier Maintenance: Changes in one area are less likely to break another.

  • Improved Reusability: Smaller, focused components are easier to reuse.

  • Better Testability: It's simpler to write unit tests for a single responsibility.
