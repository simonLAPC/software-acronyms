TDD

Test-Driven Development

Write tests before you code: create a failing test, add just enough code to pass, then refactor.

  • Ensures code quality from the start.

  • Reduces bugs by catching them earlier.

  • Acts as living documentation for your code.

Before TDD (Traditional)

Show 

1. Write a chunk of code
   Develop a large feature based on requirements.

2. Manual Testing
   Manually check if the code "looks right".

3. Write tests (maybe)
   Add tests after the fact, often to meet a metric.

4. Difficult Debugging
   Bugs are hard to trace back to their source.

5. Fear of Change
   Refactoring is risky with no safety net.

After TDD

Show 

1. Red: Write a failing test
   Write a small, automated test for new functionality.

2. Green: Make the test pass
   Write just enough code to make the test pass.

3. Refactor: Clean the code
   Clean up the code, confident the tests will catch issues.

4. Repeat Cycle
   The cycle repeats for the next small feature.

5. High Confidence
   The test suite allows confident refactoring and changes.
